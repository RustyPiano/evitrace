## 一、实验目的

针对"大模型直接问答"在情报研判场景的不足，设计并实现一个**以证据为中心**的多模态情报分析平台，验证以下能力：

1. 多模态资料（文本、PDF、图片、音频、视频）统一解析为**带来源定位**的证据；
2. 从证据中抽取人物/组织/地点/时间/事件/数量等要素并生成时间线；
3. 用**确定性规则**检测同一事件在不同来源间的时间/地点/数量冲突；
4. 生成**每条结论都可回溯到原始证据**的报告，并对引用做自动校验；
5. 在**离线/可控**前提下完成上述闭环，并支持从纯模拟到全真实模型的平滑切换。

## 二、问题背景与设计动机

直接用大模型做研判存在四个工程缺陷，本平台逐一对应解决：

| 大模型直答的问题 | 本平台的对策 |
|---|---|
| 结论无法追溯到原始资料 | 证据卡片 `E-xxxx` + 报告强制引用 + 引用自动校验 |
| 音视频与文本分析割裂 | 统一证据模型，OCR/ASR/视觉描述与文本同构进入分析链路 |
| 大模型容易"抹平"资料冲突 | 冲突检测用**纯 Python 规则**（不依赖模型主观判断），可单测可复现 |
| 缺少固定流程与可重复执行记录 | 确定性工作流 Orchestrator + 执行计划 `plan_json` + 运行记录 + 审计日志 |

## 三、系统总体设计

### 3.1 逻辑架构

```
                      Vue 3 前端（离线、无 CDN）
   登录/权限 · 任务列表 · 文件上传 · 三栏分析工作台 · 管理页(用户/Skill/健康/审计)
                              │ REST / JWT
 ┌────────────────────────────────────────────────────────────────────┐
 │                         FastAPI 后端                                  │
 │  Auth API · Task/File API · Analysis API · Admin API                 │
 │  Orchestrator（确定性工作流 + 单运行锁 + 启动恢复）                    │
 │  Skill Registry（8 个内置 Skill，可启停/健康探测）                    │
 │  模型客户端：LocalLLMClient · VisionClient · media_client(OCR/ASR)    │
 │  服务层：auth/task/storage/result/audit/run_guard …                  │
 │        ├── SQLite（8 张表）                                          │
 │        └── data/tasks/{task_id}/{original,derived,reports}/          │
 └────────────────────────────────────────────────────────────────────┘
        │ OpenAI 兼容            │ image_url 视觉        │ multipart HTTP
   本地/云 LLM(文本)        视觉 VLM(图像)        本地 OCR / ASR 服务 + ffmpeg
```

### 3.2 核心数据流（一次分析）

```
登录 → 新建任务 → 上传多模态资料 → 开始分析
  → Orchestrator 生成执行计划、加全局单运行锁、建 task_run
  → [解析 10–45%] 按模态选 Skill → 生成证据卡 E-0001…（含 locator 定位）
  → [提取 55–70%] 要素/事件/时间线（事件以 event_key 归并同一事件）
  → [冲突 80%]   纯规则比对时间/地点/数量
  → [报告 90%]   固定六段报告 + 引用校验
  → awaiting_review（100%）→ 人工审核冲突/报告 → 标记完成 → 下载 Markdown
```

### 3.3 数据模型（SQLite，8 表）

`users · tasks · task_files · task_runs · evidence · analysis_results · skill_configs · audit_logs`。
要素/事件/时间线/冲突/报告以 JSON 序列化为 Text 存储；时间统一 UTC；UUID 字符串主键；`evidence(task_id, display_id)` 唯一；`analysis_results.task_id` 唯一。

### 3.4 证据与定位器（可追溯的核心）

每条证据有任务内唯一展示号 `E-0001…`，并按模态保存定位信息 `locator_json`：

| 模态 | evidence_type | locator 关键字段 |
|---|---|---|
| 文本/PDF/DOCX | paragraph | page / paragraph / char_start / char_end |
| 图片 OCR | ocr | bbox=[x1,y1,x2,y2] |
| 图片视觉描述 | image_caption | kind=image |
| 音频 | asr | start_ms / end_ms（+说话人标签） |
| 视频音轨 | asr(video) | video_audio: start_ms / end_ms |
| 视频关键帧 OCR | video_frame_ocr | video_frame: timestamp_ms / frame_path / bbox |
| 视频关键帧描述 | video_frame_caption | video_frame: timestamp_ms / frame_path |

前端点击报告/时间线/冲突里的 `E-xxxx`，右栏即定位到原文件页码、音视频时间点或关键帧。

## 四、关键技术与创新点

1. **以证据为中心、端到端可追溯**：从原始资料 → 证据卡（带定位）→ 事件 → 冲突 → 报告，每个事实强制携带 `E-xxxx`；报告生成后用正则提取引用并校验"是否属于本任务证据 + 综合结论/时间线/主要冲突段落是否都有引用"，输出引用覆盖率与无效引用数；不满足质量门禁不允许标记完成（覆盖率<0.9 需管理员强制）。

2. **冲突检测纯规则化、可复现**：按 `event_key` 分组后两两比对——时间差 >30 分钟（或仅日期不同）判时间冲突，规范化后地点字符串不同判地点冲突，同单位数值不同判数量冲突；不调用模型，结果确定、可单元测试。这是相对"大模型主观研判"的关键区别。

3. **Skill 系统（明确输入输出 + 启停管理 + 健康探测）**：8 个内置 Skill 固定注册，管理员可启停非必需 Skill 并查看各自健康状态；区别于"Skill 只是提示词"的做法，每个 Skill 有 Manifest、Pydantic 校验的结构化输入输出。

4. **模型抽象与多档运行模式**：用 `MOCK_AI / MOCK_LLM / MOCK_MEDIA / MOCK_VISION` 四档开关与"配置即真实"的策略，实现从**纯模拟演示** → **云 LLM + 模拟媒体** → **全本地/全真实多模态**的平滑切换；业务 Skill 不直接发 HTTP，统一经 `LocalLLMClient / VisionClient / media_client`，模型输出先 Pydantic/JSON 校验再入库。

5. **跨模态视觉理解**：视频不止做关键帧 OCR（只能提字符），新增视觉理解 Skill 用支持图像的 VLM 对图片/关键帧生成**画面语义描述**，与 OCR(文字)、ASR(音频) 互补进入同一分析链路。

6. **离线与安全可控**：前端不加载任何 CDN；模型权重由部署方自备、绝不运行时自动下载；文件白名单 + 魔术字节校验 + 路径穿越防护；密钥仅来自环境变量并在日志/健康/报错中脱敏；报告标注"AI 辅助生成，需人工复核"。

## 五、实现过程（里程碑）

按 `PLAN.md` 的 6 个里程碑顺序实现，每个里程碑均"先实现、后审核、再修复、最后验收提交"：

| 里程碑 | 内容 | 关键交付 |
|---|---|---|
| M0 | 项目骨架与运行环境 | 前后端骨架、健康检查、统一错误、Docker、CWD 无关路径 |
| M1 | 登录/权限/任务/文件 | JWT 认证、8 表、任务所有权隔离、文件白名单+魔术字节+路径安全、管理员用户 API |
| M2 | 多模态解析与证据 | Skill 框架、证据服务(E-xxxx)、文档/OCR/ASR/视频解析、解析编排、证据 UI |
| M3 | Agent/提取/冲突/报告 | LLM 客户端、要素事件提取、纯规则冲突检测、报告+引用校验、Orchestrator、Analysis API |
| M4 | 完整工作台与管理页 | 三栏工作台、时间线/冲突/报告面板、用户/Skill/健康/审计页 |
| M5 | 测试/演示/部署/验收 | 单元+集成测试、3 组演示数据、对照评估脚本、Docker 生产化、README |

### 5.1 交付后扩展（应用真实模型）

在 MVP 交付后，按实际部署条件做了三轮扩展，每轮同样经过双/三方审核与真实联调：

- **接入云端文本 LLM（DeepSeek，OpenAI 兼容）**：替代"本地 LLM"槽做要素抽取与报告；修复了真实模型暴露的提示词缺陷（抽取提示词补全 JSON Schema 与示例、报告强制六段标题、裸时钟时间走 `time_text` 回退参与冲突比对）。
- **新增视觉理解 Skill（云 VLM，SiliconFlow Qwen-VL）**：图片/视频关键帧画面描述；视觉与本地 OCR/ASR 解耦（`MOCK_VISION` + "配齐 VLM 即真实"），任一失败优雅降级为 warning。
- **接入本地 OCR/ASR HTTP 服务（PaddleOCR + FunASR）**：将图片 OCR、音频/视频音轨 ASR、视频帧 OCR 从进程内库切换为调用本机 HTTP 服务；映射 box→bbox、秒→毫秒、说话人标签；服务故障非致命降级。

## 六、工程方法论：架构师统筹 + 双/三方交叉审核

本项目采用了一种可复现的协作流程，本身也是一项工程实践成果：

- **架构师（统筹）**：拆解里程碑、编写精确实现指令、做最终验证与合并决策；
- **实现**：由 Codex 按指令实现每个里程碑；
- **每阶段审核**：每个里程碑由**多个独立审核员并行交叉评审**——Opus 子代理（按维度：安全/正确性、数据模型/契约、前端/UX 等）+ Codex 独立评审；
- **最终审核**：全项目多维 Workflow（5 个维度 + 对抗性验证）+ Codex 全局评审 + 现场实测三路并行。

该流程在每个阶段都发现了**真实且互补**的严重缺陷（单一审核者难以全部覆盖），例如：

| 阶段 | 发现者 | 典型严重缺陷（已修复） |
|---|---|---|
| M0 | Codex / Opus | `CORS_ORIGINS` 作环境变量触发 pydantic JSON 解码崩溃；数据路径依赖 CWD；异常 `str(exc)` 外泄 |
| M1 | 三方 | 默认 `SECRET_KEY=change-me` 实际生效可伪造 admin JWT；MIME 校验可被 octet-stream 绕过；完成状态机缺校验 |
| M2 | Codex | 无音轨视频不再抽帧；MOCK 显式空结果回退默认 → 伪造证据；真实模型默认会联网下载（违反离线） |
| M3 | Codex | **Python 3.11(Docker) 启动崩溃**：注解引用未导入名（本机 3.14 因惰性注解掩盖）；`datetime` 是 `date` 子类致"同日"误判时间冲突 |
| M4 | Codex | >50 条证据时 `[E-xxxx]` 引用联动失效（分页上限 50）；Skill 健康错误泄露模型目录绝对路径 |
| 终审 | Workflow | 解析错误信息未脱敏（真实模式路径泄露回归） |

**结论**：跨模型、跨环境（本机 3.14 vs 部署 3.11）、跨视角的交叉审核，是发现"只在真实/部署环境暴露"缺陷的有效手段。

## 七、实验与验证

### 7.1 自动化测试

- 后端 `pytest`：**192 个用例全部通过**（24 个测试文件，覆盖文件类型/路径安全、证据编号、时间/地点/数量冲突、引用校验、权限依赖、MOCK 输出校验、各模型客户端、HTTP 媒体适配、健康检查等）。
- 三类冲突均有独立单测，含边界用例（恰好 30 分钟不冲突、跨午夜、仅日期、纯日期 vs 同日时刻、裸时钟时间、时区不一致跳过）。
- 集成测试覆盖完整业务流（两名分析员 + 越权 404 + 管理员可访问 + 改冲突状态 + 下载报告）。
- 前端 `npm run type-check && build` 通过；`docker compose config` 合法；Python 3.11 容器内 `import app.main` 成功。

### 7.2 演示数据对照评估（MOCK 模式，确定性可复现）

`scripts/build_demo_data.py` 生成 3 组虚构案例（时间/地点/数量冲突，跨模态植入），`scripts/evaluate_demo.py` 走**真实分析管线**（解析→证据→规则冲突→引用校验→报告，仅模型环节用确定性 fixture）后与 `expected.json` 对照：

| 案例 | 预期冲突 | 发现冲突 | 冲突召回率 | 报告引用 | 有效引用 | 引用覆盖率 | 无效引用 | 四模态齐全 |
|---|---:|---:|---:|---:|---:|---:|---:|---|
| case_01 时间冲突 | 1 | 1 | 1.00 | 6 | 6 | 1.00 | 0 | 是 |
| case_02 地点冲突 | 1 | 1 | 1.00 | 6 | 6 | 1.00 | 0 | 是 |
| case_03 数量冲突 | 1 | 1 | 1.00 | 5 | 5 | 1.00 | 0 | 是 |

对照 `SPEC.md §15.2` 质量验收门槛：引用覆盖率 ≥0.90（实测 **1.00**）、无效引用 ==0（实测 **0**）、植入冲突发现 ≥80%（实测 **100%**）—— 全部达标。

> 评估脚本经独立审核确认**未"对答案打分"**：冲突/覆盖率/无效引用均由真实管线计算，`expected.json` 仅用于运行后比对，且使用临时隔离数据库。

### 7.3 真实多模态端到端联调（全部使用真实模型/服务）

在一个任务中同时接入全部真实模型，验证完整闭环：

- 文本 `brief.txt`（真实文档解析）："第三装甲连于 14:00 抵达海港门，车辆 3 辆"
- 图片（真实 **PaddleOCR** `:8000`）：OCR 出 4 个文本框（含 bbox），"Delta check-in 14:00 … 3 vehicles"
- 图片（真实 **VLM** SiliconFlow Qwen-VL）：生成整图场景描述
- 音频（真实 **FunASR** `:8001`，真实语音）：转写"…下午四点三十分抵达海港门…随行车辆五辆"，带秒级时间戳与说话人标签 `[说话人1]`
- 要素抽取与报告（真实 **DeepSeek**）：将文本/图片/音频证据归并到同一事件

结果：规则引擎检出**跨模态时间冲突（14:00 ↔ 16:30）与数量冲突（3 辆 ↔ 5 辆）**，左右两侧分别引用文本证据与音频证据；引用覆盖率 1.00、无效引用 0、**零 warning**、真实模型报告（非模板降级）。证明"证据可追溯 + 跨模态冲突检测"在真实模型下成立。

### 7.4 部署验证（Docker / Python 3.11）

`docker compose up --build -d` 后端/前端容器均 healthy；容器内 `import app.main` 成功（验证此前修复的 3.11 注解崩溃）；经 Nginx 走完整 MOCK 闭环（登录→上传→分析→报告下载）；`docker compose restart` 后数据/报告保留（NFR-003 可恢复）。

## 八、结果分析与讨论

1. **可追溯性可量化达成**：引用覆盖率 1.00、无效引用 0，说明"每条结论可回溯到证据"不是口号而是被校验的硬约束。
2. **冲突检测的价值**：在真实模型下，文本与语音对同一事件给出 14:00/3 辆 与 16:30/5 辆 两套说法，平台**显式标出冲突并各自给出来源**，而非由模型"二选一/抹平"——这正是情报研判最需要的能力。
3. **与"基线直答"的对照（设计层面）**：基线做法（把全部文本一次性丢给模型直接出报告）无法保证引用有效性、不显式暴露冲突、且难以纳入图片/音视频；平台做法以可校验的结构化中间产物（证据/事件/冲突）为骨架，把模型的"自由发挥"约束在可追溯范围内。注：本报告对平台组给出了量化指标，基线组的量化对照可作为后续工作（评估脚本预留了对照位）。
4. **真实模型暴露的工程要点**：MOCK 充分但不足以替代真实联调——抽取提示词必须显式给出 Schema、推理型模型需读 `content` 且不设过小 `max_tokens`、裸时钟时间需回退比对、跨语言实体（"第三装甲连" vs "Delta"）会影响 `event_key` 归并。这些都在真实联调中被发现并处理或记录。

## 九、安全与合规边界

- 仅用于授权/公开/虚构数据的离线辅助分析；不含自动目标识别、行动决策、武器控制或自主指挥功能；报告强制标注需人工复核。
- 离线：前端无 CDN、模型权重自备不自动下载、核心流程不连公网（云模型为用户显式配置）。
- 安全：JWT + 任务所有权隔离（越权一致返回 404）；文件白名单 + 魔术字节 + 路径穿越防护 + UUID 落盘；密钥仅来自环境变量并在日志/健康/报错脱敏；管理操作全审计；容器 `read_only` + tmpfs、不打入模型权重。

## 十、已知限制与未来工作

1. 视觉理解为**关键帧画面描述**，非时序动作识别/目标跟踪；视频理解仍是抽帧粒度。
2. 跨语言/跨别名实体未做消歧（"第三装甲连"与"Delta"不会自动归并为同一事件），依赖证据用语一致或后续增加别名/对齐。
3. 基线组（一次性直答）的量化对照未纳入本次结果，可作为后续对照实验。
4. 系统截图为占位，需在演示时补充。
5. 容器以 root 运行（已 `read_only`+tmpfs；非 root + cap_drop 因 bind-mount/nginx:80 端口绑定耦合而暂缓）。
6. FunASR 对静音返回空分段，真实演示需准备真实语音音频。

## 十一、结论

本实验完整实现并验证了一个**以证据为中心、可追溯、可离线部署**的多模态情报分析平台：

- 功能闭环（登录→多模态资料→可定位证据→要素/时间线→规则冲突→带引用报告→人工复核）全部跑通；
- 质量指标达到并超过规格门槛（引用覆盖率 1.00、无效引用 0、植入冲突召回 100%）；
- 在**全真实模型**（DeepSeek 文本 + Qwen-VL 视觉 + PaddleOCR + FunASR + ffmpeg）下完成跨模态冲突检测的端到端验证；
- 可通过 Docker 在单机离线环境部署，重启数据可恢复。

平台有效缓解了"大模型直答"在情报研判中的不可追溯、跨模态割裂、易抹平冲突、缺乏固定流程等问题，达到课程 MVP 的设计目标。
