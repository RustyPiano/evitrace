## M1-01 数据模型（SPEC §7）
在 `app/models.py` 用 SQLAlchemy 2 实现 8 张表，列名/类型/约束严格对照 SPEC §7：
`users, tasks, task_files, task_runs, evidence, analysis_results, skill_configs, audit_logs`。
要求：
- 主键 UUID 用字符串（`str`，默认值用 `uuid4().hex` 或带连字符均可，全局统一）。
- 所有时间字段统一存 UTC（用 timezone-aware 的 `datetime.now(timezone.utc)`，不要用已弃用的 `utcnow`）。
- 所有 JSON 字段先序列化为 Text 存储（`plan_json/warnings_json/locator_json/entities_json/events_json/timeline_json/conflicts_json/citation_check_json/detail_json` 等）。
- 建立必要索引与约束：`users.username` unique；`evidence (task_id, display_id)` 唯一索引；`analysis_results.task_id` unique；常用外键列加索引（task_files.task_id、evidence.task_id 等）。
- 状态枚举用字符串常量（不要 Python Enum 列类型也可，但取值必须等于 SPEC §4.2 / §5.3 的字面值）。提供集中定义的状态常量模块（如 `app/constants.py`）：Task 状态、文件状态、run 状态、冲突状态。
- 启动时根据 Skill registry 初始化 `skill_configs`：在 `app/skills/registry.py` 定义 7 个 Skill 的 `SkillManifest`（仅声明元数据即可，`run()` 实现留到 M2）。**required 标记遵循 SPEC §5.9 SKILL-004 的行为规则：仅 `intelligence_extract`、`conflict_detect`、`report_generate` 为 required（不可停用）；`document_parse/image_ocr/audio_transcribe/video_parse` 为非必需（可停用，停用后对应文件标记 warning）。** （SKILL-002 里 document_parse required=True 只是 manifest 结构示例，以 SKILL-004 行为为准；在最终消息里说明此决定。）启动时 upsert 到 `skill_configs`（已存在则不覆盖管理员的 enabled 设置）。

级联删除：用**服务层**实现（不靠 DB 外键 ON DELETE）。删除任务时删除其 task_files / task_runs / evidence / analysis_results / 相关 audit 处理按需，并删除磁盘目录 `data/tasks/{task_id}`。

## M1-02 认证（SPEC §5.1 §8.1 §13）
文件：`app/services/auth_service.py`、`app/api/auth.py`、`app/dependencies.py`、`app/services/audit_service.py`。
- 密码哈希存储（不可逆）。**确保在本机 Python 3.14 上哈希与校验实际可用并写测试验证**；passlib+bcrypt 若有兼容问题，可改用 `bcrypt` 库直连或 `pwdlib[bcrypt]`。在 requirements 增加所选依赖。
- JWT 创建与校验（建议 PyJWT，简单稳定）；`SECRET_KEY` 来自 settings，过期用 `ACCESS_TOKEN_EXPIRE_HOURS`。
- 依赖：`get_current_user`（校验 JWT、查用户、停用用户拒绝）、`require_admin`。
- 启动时 seed 默认管理员（`FIRST_ADMIN_USERNAME/PASSWORD`），已存在则跳过。
- `POST /api/v1/auth/login`：成功返回 `{access_token, token_type:"bearer", user:{id,username,role}}`；用户名或密码错误返回 401 `INVALID_CREDENTIALS`（不区分是哪一项）；停用用户返回 403 `INACTIVE_USER`。
- `GET /api/v1/auth/me`：返回 `{id, username, role, is_active}`。
- 审计：登录成功/失败写 `audit_logs`（action 如 `login_success`/`login_failed`）。不要记录密码/JWT/明文敏感内容（NFR-005）。

## M1-03 任务 API（SPEC §5.2 §8.2 §13）
文件：`app/api/tasks.py`、`app/services/task_service.py`。
- `GET /api/v1/tasks`：列表。分析员只见自己的（`owner_id == current_user.id`），管理员见全部。返回字段满足 SPEC §5.2 TASK-002（名称、创建人、文件数、状态、最近更新时间、最近运行错误摘要）。
- `POST /api/v1/tasks`：创建。字段校验：`name` 1–100、`objective` 1–1000、`description` ≤2000。创建后 `status=draft`。
- `GET /api/v1/tasks/{task_id}`：owner/admin。非 owner 且非 admin 一律返回 `TASK_NOT_FOUND`（404），**保持一致**（不要有时 403 有时 404，避免越权探测）。
- `PATCH /api/v1/tasks/{task_id}`：owner/admin 改基本信息或标记完成（M1 先支持改 name/objective/description；标记完成的状态机校验可留占位，真正分析状态在 M3）。
- `DELETE /api/v1/tasks/{task_id}`：owner/admin。运行中（status 属于 queued/parsing/extracting/detecting_conflicts/generating_report）禁止删除并明确报错；删除时走服务层级联删除 + 删除磁盘目录；写审计日志。
- 审计：创建任务、删除任务写 audit。

## M1-04 文件存储服务（SPEC §5.3 §8.3 §12 §13）
文件：`app/services/storage_service.py`、`app/api/files.py`、`app/utils/file_types.py`。
- 支持格式白名单（FILE-001）：txt, md, pdf, docx, jpg, jpeg, png, wav, mp3, m4a, mp4。按扩展名映射 modality：text(txt/md)、document(pdf/docx)、image、audio、video。
- 双重校验：扩展名 + MIME（可用 `mimetypes` 或读取文件头/`python-magic` 不强制；至少扩展名严格白名单 + 基础 MIME 检查）。拒绝 `.exe/.sh/.bat/.js` 等可执行/脚本，拒绝双扩展名绕过（取最终扩展名严格校验）→ 400 `FILE_TYPE_NOT_SUPPORTED`。
- 大小限制：`MAX_UPLOAD_MB`，超限 413 `FILE_TOO_LARGE`（注意流式读取时累计大小，避免把超大文件整体读入内存）。
- 磁盘文件名用 UUID（`{uuid}.{ext}`），原始文件名清洗后单独存 `original_name`。任务目录结构按 SPEC §6.3：`data/tasks/{task_id}/original/`、`derived/{extracted,audio,frames,thumbnails}/`、`reports/`（M1 至少建 original/，其余目录可在需要时创建）。
- **路径安全**：任何接口不得接受用户磁盘路径；构造路径后用 `Path.resolve()` 校验仍在任务目录内，拒绝 `../` 穿越。
- 接口：`GET /tasks/{task_id}/files` 列表；`POST /tasks/{task_id}/files` multipart 多文件上传（owner/admin）；`DELETE /files/{file_id}`（仅未运行任务，owner/admin）；`GET /files/{file_id}/stream` 受权限保护的原文件流（支持媒体播放，建议支持 Range 以便音视频跳转）；`GET /files/{file_id}/preview`（M1 可先返回基础信息/占位，真正文本预览在 M2）。所有文件接口必须先校验所属任务的 owner/admin 权限，无权返回 404/403 一致策略。
- 审计：上传文件写 audit。
