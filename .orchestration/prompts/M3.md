## 关键架构约束
- **业务 Skill 不得直接调用 requests/httpx**；所有模型调用经 `LocalLLMClient`。
- **任何模型输出先 Pydantic 校验再写库**；不得写入未验证 JSON。
- 不把二进制/原始文件交给模型；提取只传证据文本摘要（SPEC §9.2/§9.4）。
- 冲突检测是**纯 Python 规则**，不调用模型。
- 严格命名：状态枚举、错误码、event/conflict 结构字段、API 路径与 SPEC 一致。

## M3-01 本地模型客户端 `app/services/llm_client.py` + `app/utils/json_repair.py`
- `LocalLLMClient`：`generate_json(system_prompt, user_prompt, schema: type[BaseModel]) -> BaseModel` 与 `generate_text(system_prompt, user_prompt) -> str`。
- 真实模式：OpenAI-compatible Chat Completions（`LOCAL_LLM_BASE_URL/API_KEY/MODEL`），用 httpx，超时 `LLM_TIMEOUT_SEC`，最多 `LLM_MAX_RETRIES`(2) 次重试。
- JSON 提取与轻量修复（`json_repair`）：去除 ```code fence```、提取首个 JSON 对象、轻修复尾逗号等；再用 schema 校验；解析/校验失败则重试；最终失败抛 `INVALID_MODEL_OUTPUT`(500)。
- 模型不可达（连接错误/超时且重试耗尽）抛 `LOCAL_MODEL_UNAVAILABLE`(503)。
- 记录 token/耗时指标（若接口返回），日志含简要信息但不含敏感全文（NFR-005）。
- `MOCK_AI=true`：不联网；`generate_json`/`generate_text` 由调用方传入的 mock 生成器或在 client 层返回确定性占位（建议：client 提供 `mock_json`/`mock_text` 回调参数，使各 skill 控制各自 MOCK 产出）。
- 健康检查函数：ping 模型可用性（MOCK 下返回 healthy/skipped）。
测试：MOCK 输出稳定确定；真实模式非 JSON 输出触发重试（可用可注入的假 transport/mock）；最终失败返回 INVALID_MODEL_OUTPUT；不可达返回 503。

## M3-02 要素与事件 Schema `app/schemas_analysis.py`（或并入 schemas）
Pydantic 模型：`Entity`（type∈person/organization/location/event/object/time/quantity、name、可选 confidence/evidence_ids）、`Quantity`(value/unit)、`Event`（结构见 SPEC §5.6 ANALYSIS-002：event_id/event_key/title/subject/action/object/time_text/time_normalized/location/quantity/evidence_ids/confidence）、`ExtractionResult`(entities/events)、`TimelineItem`、`Conflict`（见 §5.7）、`CitationCheck`(citation_coverage/invalid_citations/...)。
要求：`confidence` ∈[0,1]；`evidence_ids` 至少 1 个；`event_id` 由服务端统一重编号（EVT-001…）；引用不存在的事件**过滤并告警**或拒绝保存（择一，且要在 extraction 阶段对 evidence_ids 做存在性校验，剔除无效引用、若导致 evidence_ids 空则丢弃该事件并 warning）。

## M3-03 Intelligence Extract Skill `app/skills/intelligence_extract.py`
- 真实模式：按证据分批（每批约 20–30 条，单批 ≤ ~12000 中文字符，SPEC §9.2）构造 prompt（证据格式 `[E-0001][来源文件][定位] 内容`），系统提示词满足 §9.3（只用输入证据、不确定输出 null、每事件须引用证据、不出行动建议、不合并明显冲突事实、严格 JSON schema）；逐批 `generate_json(ExtractionResult)`；跨批合并：实体按 (type,name 规范化) 去重；事件按 `event_key + 证据来源` 保留。
- 生成时间线：有 `time_normalized` 的事件按时间升序；模糊时间单独分组（“时间未确定”）。
- 简化策略（SPEC §5.6 / PLAN）：不做高级实体消歧；仅 trim、大小写统一、全半角统一；时间解析失败保留 `time_text`、`time_normalized=null`。
- **MOCK_AI 行为（确定性）**：
  - 若存在任务级 fixture `data/tasks/{task_id}/mock/extraction.json`，读取并使用（其 evidence 引用可用 display_id 或按内容子串 `match` 字段解析为实际 display_id）；这是 M5 演示用例注入植入冲突的机制。
  - 否则使用**确定性默认提取**：基于现有证据生成有效结果——至少产出包含可演示冲突的事件（例如对存在的前若干条证据，生成两个同 `event_key`、`time_normalized` 相差>30 分钟的事件，引用真实存在的 display_id），保证端到端 MOCK 能产出时间线与至少一个冲突，且所有 evidence_ids 真实有效。
- 保存 `entities_json/events_json/timeline_json` 到 analysis_results（由 orchestrator 统一保存亦可）。
- 证据为空时明确失败（不伪造）。
测试：每事件有有效证据；时间线排序正确；模型返回未知字段不致崩（schema 容错/忽略额外字段）；证据为空明确失败；MOCK 确定性。

## M3-04 冲突检测 Skill `app/skills/conflict_detect.py` + `app/utils/time_normalize.py`（纯 Python，不调模型）
按 `event_key` 分组，组内两两比较：
- **时间**：两个可解析的精确时间差 > `TIME_CONFLICT_MINUTES`(30) 视为冲突；只有日期时，日期不同即冲突；无法解析的不比较。
- **地点**：trim、移除常见空格与标点后比较；完全相同视为一致；不同且均非空 → 冲突；不实现地名别名库。
- **数量**：单位相同且数值不同 → 冲突；单位不同 → 不自动判冲突，只记 warning。
- 每个冲突含双方事件与证据编号；`conflict_id` 形如 `C-001`；`status` 默认 `unreviewed`。结构严格符合 SPEC §5.7。
必须包含的单元测试（纯函数、无 DB 依赖）：14:00 vs 16:30→时间冲突；14:00 vs 14:10→非冲突；地点 A vs B→地点冲突；3 辆 vs 5 辆→数量冲突；3 辆 vs 3 人→不比较；不同 event_key→不比较。

## M3-05 报告生成与引用验证 `app/skills/report_generate.py` + `app/utils/citations.py`
1. 固定 Markdown 结构（SPEC §5.8 REPORT-001 六个章节：一、任务概述/二、资料概况/三、事件时间线/四、主要冲突/五、综合分析结论/六、未确认事项）。报告须含醒目「AI 辅助生成，需人工复核」声明（§12）。
2. 调用模型生成正文（真实模式 `generate_text`，输入仅任务目标/文件摘要/事件/时间线/冲突/必要证据短文本，§9.4；不得新增事实，事实性陈述须带 `[E-xxxx]`）。
3. 引用验证（`citations.py`，按 PLAN 8.3）：正则 `E-\d{4}` 提取；校验属于当前任务证据；统计无效引用；计算「五、综合分析结论」每个非空段落是否至少一个引用 → `citation_coverage`；输出 `citation_check`（invalid_citations/citation_coverage/...）。
4. 保存报告与 citation_check；写 `data/tasks/{id}/reports/latest.md`。
5. **模型失败 fallback（MUST）**：模型生成失败也用纯模板生成最小报告（任务概述、文件统计、时间线列表、冲突列表、“综合结论生成失败，请人工复核”），任务仍可进入 `awaiting_review` 但带 warning。
- **MOCK_AI 行为**：生成固定模板报告，**只引用真实存在的 display_id**，且“综合分析结论”每个非空段落都含引用，使 `invalid_citations==0` 且 `citation_coverage>=0.9`（满足验收 §15.2）。
测试：无效引用能被发现；覆盖率计算正确；报告文件可下载；无模型时 fallback 可用；MOCK 报告 invalid=0 且 coverage≥0.9。

## M3-06 Orchestrator `app/services/orchestrator.py`
职责（参考 PLAN 8.1 伪代码）：
- 校验任务可运行（有可分析文件，否则 `TASK_NOT_READY`(409)；required skill 被停用则 `REQUIRED_SKILL_UNAVAILABLE`(503) 并禁止启动）。
- 用 `run_guard` 保证**单机一次只运行一个任务**（已运行 → `TASK_ALREADY_RUNNING`(409)）。
- 生成 `plan_json`（SPEC §5.5 AGENT-002 结构），创建 `task_run`（status=running）。
- 依次执行：`parse_all_files`（复用 M2 核心）→ `intelligence_extract` → `conflict_detect` → `report_generate`(含 citation_validate)。
- 每步更新 `Task.status`/`task_run.progress`/`current_step`；进度建议：queued 0、parsing 10–45、extracting 55–70、detecting_conflicts 80、generating_report 90、awaiting_review 100（进度单调不减）。
- 解析后若无证据 → 明确失败（`ANALYSIS_FAILED`）。
- 捕获异常：run→failed、task→failed、保存安全错误摘要（不泄露内部细节，NFR-005）；汇总 warnings 到 run.warnings_json。
- 成功后 task→`awaiting_review`，保存 AnalysisResult（entities/events/timeline/conflicts/report/citation_check）。
- **重跑逻辑**：保留原文件；删除旧 evidence；清空旧 analysis_result；新建 run；不覆盖历史 task_run。
- **启动恢复（NFR-003）**：应用启动时把 DB 中仍为 running 的 run 标 failed、对应 task.status=failed、last_error=“服务重启导致运行中断，请重新执行”。挂到 lifespan startup。
- 日志含 task_id/run_id（NFR-005）。
测试：完整 MOCK 流程完成并 awaiting_review；进度单调增加；运行中再次启动返回 409；失败后可重跑；启动恢复把 running 置 failed。

## M3-07 Analysis API `app/api/analysis.py`（owner/admin 权限，复用归属校验）
- `POST /api/v1/tasks/{task_id}/runs`：启动或重新分析（BackgroundTasks 触发 orchestrator，遵守 run_guard；无文件 409 `TASK_NOT_READY`；已运行 409 `TASK_ALREADY_RUNNING`）。返回 `{run_id,status:"queued"}`。
- `GET /api/v1/tasks/{task_id}/runs/latest`：最新运行状态（run_id/status/progress/current_step/warnings）。
- `GET /api/v1/tasks/{task_id}/results`：全部结构化结果（entities/events/timeline/conflicts/report/citation_check 概要）。
- `PATCH /api/v1/tasks/{task_id}/conflicts/{conflict_id}`：修改冲突审核状态（unreviewed/confirmed/ignored），更新 conflicts_json 并持久化。
- `POST /api/v1/tasks/{task_id}/report/regenerate`：基于现有结构化结果重生成报告（不重跑解析/提取）。
- `GET /api/v1/tasks/{task_id}/report/download`：下载 Markdown（文件名 `任务名_分析报告_YYYYMMDD_HHmm.md`，§5.8 REPORT-004）。
（evidence list/detail/source/frame 已在 M2，不重复。）
测试：权限覆盖全部接口；无文件 409；运行中再次运行 409；非 owner 不能查看结果/下载。
