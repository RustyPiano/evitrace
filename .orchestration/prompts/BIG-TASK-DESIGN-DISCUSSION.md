## 项目背景（EviTrace，课程作业级 MVP，但要尽量工程化）
多模态情报分析平台。流水线：**解析（OCR/ASR/文档→证据卡片）→ 要素抽取（LLM）→ 冲突检测（规则引擎）→ 报告生成（LLM）**。
- 后端：FastAPI + SQLAlchemy + **SQLite**；真实分析跑在 **FastAPI BackgroundTasks（同进程线程）**，有**单运行锁**（一次只允许一个分析）。前端 Vue 轮询 `/runs/latest` 看进度。
- 真实 LLM：OpenAI 兼容接口，**用户现在用 OpenRouter**（之前 DeepSeek 直连）。抽取是**每批一次 `chat/completions` 调用**。
- 抽取实现：把证据切批（默认 30 条/批 或 12000 字符/批），**有界并发**（`EXTRACT_CONCURRENCY` 默认 4）跑批，主线程收集、合并、分配 `event_id`；进度 55%→70% 按批推进；可协作式取消。
- 每批都重发约 1800 token 的固定 system 提示词（JSON schema + 两个完整 few-shot 示例）。

## 痛点（要解决的核心问题）
真实跑大语料时体验极差。实测：**61 个文档 → 8121 条证据 → 约 271 批 LLM 调用**。表现：
1. **成本高**：成本随证据条数线性增长，一次 ≈ 240 万输入 token；用户持续被扣费且肉疼。
2. **慢**：271 批，wall-clock 很长。
3. **被限流**：上游返回 **HTTP 429（Too Many Requests）**。并发 4 把请求打太快就触发。
4. **脆弱 / 全有或全无**：当前一旦某批最终失败（429 重试耗尽/超时），`future.result()` 抛异常 → 整个 `_run_real_extraction` 失败 → 整轮 run 标记 failed，**前面已完成的 200+ 批结果全部丢弃**，重跑又从头来（又烧一遍钱、又撞限流）。
5. **重试太糙**：LLM client 对 429 当作「不可用」**立即重试 3 次、无退避、无视 Retry-After** → 三次几毫秒内全撞 429，等于没重试。
6. **UX 差**：跑完才出结果（中途看不到任何产出），失败后前端还在轮询 `/results` 刷一堆 404。
7. 没有**断点续跑 / 增量缓存 / 部分结果持久化**。

## 已经做过的缓解（请在此基础上讨论，别重复造）
- 抽取前**精确去重 + 空白/过短过滤**（仅减少发给 LLM 的证据，全量证据仍入库）。
- **批量大小可配**（调大→批数少→重复提示词税少）。
- **启动护栏**：文件数超阈值需二次确认（防误传大语料）。
- **有界并发 + 可取消 + 按批进度**。
- 关键缺口仍在：**429 无退避、无断点续跑、无部分持久化、全有或全无、跑完才出结果**。

## 我已想到的方向（请批判 + 排序 + 补充，并指出对课程级 MVP 哪些过度工程）
A. **韧性/不丢进度**：每批结果**即时持久化**到 DB（批级状态表：pending/done/failed + 结果），失败不再炸全局、**优雅降级出部分结果**；run 可**断点续跑**（只重跑 failed/pending 批）。
B. **限流处理**：429 **指数退避 + 读取 Retry-After**；**自适应并发/令牌桶节流**（按上游 RPM/TPM 收敛）；OpenRouter 的限流头利用。
C. **从源头砍量（根因）**：**按任务目标做相关性预筛/检索**（embedding 或廉价启发式先选 top-K 相关证据，只对相关的抽取）——大部分 8121 证据可能与具体目标无关；语义近重复合并；map-reduce/层次化。
D. **作业模型**：是否要从同进程 BackgroundTasks 升级到**真正的任务队列**（Celery/RQ/arq + Redis），还是在 SQLite 内做**轻量批级队列 + worker 循环 + 断点续跑**就够（避免给课程项目引入 Redis 等重依赖）？
E. **UX**：**流式/增量出结果**（每批完成即增量展示实体/事件）；进度带 ETA + 已用成本估算 + 失败批数；失败可一键续跑；停止后保留已得部分。
F. **模型分层/缓存**：bulk 抽取用便宜模型、合成用强模型；OpenRouter 的 prompt caching / provider 路由；同语料重跑的**抽取结果缓存**（按内容哈希）。
