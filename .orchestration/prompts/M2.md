## 关键架构约束（务必遵守）
1. **重型依赖延迟导入**：PyMuPDF、python-docx、charset-normalizer、Pillow、PaddleOCR、faster-whisper、FFmpeg(subprocess) 等**只能在对应 Skill 的执行路径内 import**，不得在模块顶层/应用启动路径 import。保证无这些依赖时应用仍能启动、测试仍能跑。
2. **依赖分层**：
   - 轻量解析依赖（PyMuPDF、python-docx、charset-normalizer、Pillow）加入 `backend/requirements.txt`，并**实际安装到 `backend/.venv` 并验证可用**（document_parse 真实工作，不依赖模型）。
   - 重型 ML 依赖（paddleocr、faster-whisper 及其后端）放入单独的 `backend/requirements-optional.txt`（真实 OCR/ASR/视频模式才需要），**不要装进 .venv**（体积大、本环境跑 MOCK 演示）。README 说明真实模式如何安装。
3. **MOCK_AI（默认 true）行为**：
   - `document_parse`：始终**真实**解析（PyMuPDF/docx/charset-normalizer 是确定性的，无需模型），即使 MOCK_AI=true 也真实读取文本与页码。
   - `image_ocr` / `audio_transcribe` / `video_parse`：MOCK_AI=true 时**不加载真实模型**，改为读取 fixture。fixture 机制：先查原文件同任务下的 sidecar fixture（约定：`data/tasks/{task_id}/original/{stored_name}.mock.json` 或按 original_name 映射的 `*.ocr.json/*.asr.json`），找不到则用**内置默认确定性 fixture**（保证总能产出结构正确的证据：OCR 至少一条带 bbox 的证据、ASR 至少 2 段带 start_ms/end_ms 的证据、video 至少 1 段音轨证据 + 1 帧 OCR 证据，并用 Pillow 生成一张占位关键帧 PNG 写入 `derived/frames/`）。
   - MOCK_AI=false 时才走真实模型（延迟加载 + 进程内缓存）。
4. 严格命名：locator `kind` 取值、evidence_type、modality 必须与 SPEC §5.4/§7 一致（`text/image/audio/video`；evidence_type `paragraph/ocr/asr/video_frame_ocr`；video 音轨 locator kind `video_audio`、关键帧 `video_frame`）。
5. Skill 只能读写当前任务目录；路径用 M1 的安全校验（resolve + 任务目录前缀）。

## M2-02 证据服务 `app/services/result_service.py`
- `next_display_id(task_id)`：查询任务内最大数字后缀，返回 `E-%04d`（从 E-0001 起递增）。
- 批量写入证据：接收 skill 产出的证据项列表（content/modality/evidence_type/locator(dict)/confidence/file_id/skill_id），统一分配递增 display_id（同任务不重复），locator 序列化为 Text 存库。
- 删除某文件的旧证据（重解析前调用）。
- 分页查询任务证据（默认 50/页，SPEC NFR-004），支持按 file_id/ modality 过滤。
- locator 序列化/反序列化工具。
- 证据 source 响应：给定 evidence，返回 locator + 受权限保护的文件 URL（原文件用 M1 的 `/files/{id}/stream`；视频关键帧图片需可访问——见下）。
测试：display_id 从 E-0001 递增、单任务不重复；证据只能被任务 owner/admin 查询。

## M2-03 文档解析 Skill `app/skills/document_parse.py`
- TXT/MD：用 charset-normalizer 检测编码读取；按空行切分段落（SPEC EVID-001）。
- DOCX：python-docx 读非空段落。
- PDF：PyMuPDF 按页提取文本，再按**最多 1000 字符**切块。
- 每段/块一条证据，locator 含 page/paragraph/char_start/char_end（按模态填充，缺的置 null）。空白块不保存。
- 扫描型 PDF 无文本：标记 warning（首版不做整页 OCR）。
- 损坏文件：捕获异常，标记该文件 failed，不抛未捕获异常。
fixture：在 `backend/tests/fixtures/` 准备 1 个 txt、1 个 docx、1 个两页 pdf（可用 PyMuPDF 程序化生成两页 PDF）。
测试：证据内容正确、PDF 页码正确、空段落不保存、损坏文件标记 failed 不崩。

## M2-04 图片 OCR Skill `app/skills/image_ocr.py`
- 封装 PaddleOCR：MOCK_AI=false 时延迟加载+缓存；MOCK_AI=true 读 fixture/默认 fixture。
- 每个文本框一条证据，locator `{kind:"image", bbox:[x1,y1,x2,y2]}`，evidence_type=`ocr`。
- 无文本返回 warning，不伪造内容。
测试（MOCK 模式）：含文字图片(默认fixture)至少一条证据、locator 有 bbox、无文字情形返回 warning 不伪造。

## M2-05 音频转写 Skill `app/skills/audio_transcribe.py`
- 封装 faster-whisper：真实模式延迟加载；MOCK 读 fixture/默认 fixture。
- 分段输出文本+时间，locator `{kind:"audio", start_ms, end_ms}`，evidence_type=`asr`，过滤空分段，支持 WAV/MP3/M4A。
- 无语音返回 warning。
测试（MOCK）：证据 start_ms/end_ms 正确、点击定位所需字段完整、无语音返回 warning。

## M2-06 视频解析 Skill `app/skills/video_parse.py`
- 真实模式：检查 FFmpeg 存在；提取 WAV 音轨→复用音频转写逻辑（但 evidence modality=`video`、locator kind=`video_audio`）；每 `VIDEO_FRAME_INTERVAL_SEC`(默认10s) 抽帧→对帧 OCR（locator kind=`video_frame`，含 timestamp_ms/frame_path/bbox）。派生文件写 `derived/audio`、`derived/frames`。FFmpeg 缺失时错误明确（不崩进程）。
- MOCK 模式：用 fixture/默认 fixture 产出音轨证据 + 关键帧 OCR 证据；用 Pillow 生成占位关键帧 PNG 写入 `derived/frames/` 并在 locator.frame_path 引用（相对任务目录）。
- frame_path 只能位于任务目录内（路径安全校验）。不实现动作识别。
测试（MOCK）：短视频生成音轨或帧证据、frame_path 在任务目录内、无音轨仍可抽帧（MOCK 下等价于仍产帧证据）。

## M2-07 文件解析编排 `app/services/parse_service.py`
- `parse_all_files(task_id, run_id=None) -> ParseSummary`：按文件 modality 选择 skill（text/document→document_parse；image→image_ocr；audio→audio_transcribe；video→video_parse）。
- 每文件：状态 uploaded→parsing；解析前删除该文件旧 evidence；调用 skill；写证据；成功→parsed，有 warning→warning，失败→failed（记 error_message）。**单文件失败不影响其他文件**。
- 跳过被停用的解析 skill：对应文件标记 warning。
- 汇总 warnings；任务状态在解析期间置 `parsing`（最终状态由 M3 编排决定，此处 M2 可在解析后置回合适状态或保留，供 M3 复用——本函数要可被 M3 orchestrator 直接复用，不要写死任务最终状态）。
- 进度按文件数更新（如有 run_id 则更新 task_run.progress）。
**为使 M2 可独立演示**，新增 `POST /api/v1/tasks/{task_id}/parse`（owner/admin，用 BackgroundTasks 触发 `parse_all_files`，立即返回；运行中冲突返回 409），以及在 M3 中 orchestrator 复用同一 `parse_all_files`（不要写两份解析逻辑）。
验收：混合上传 5 类文件可运行；至少一种文件故障时其他证据仍生成；文件状态与 warning 可被前端读取。
